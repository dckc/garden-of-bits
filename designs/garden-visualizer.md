# Garden Visualizer

A lightweight web server that renders the garden's live and historical
activity in a browser. Intended for the maintainer to open, glance at, and
understand what the garden is doing right now and what it did recently.

## Data sources

The visualizer reads five sources. Each is independent; if one is
unavailable, the remaining views degrade gracefully.

### Source A: Journal branch (git)

The journal at `journal/entries/<YYYY>/<MM>/<DD>/<HHMMSS>Z-<kind>-<role>-<short-id>.md`
is the garden's append-only transcript. Every dispatch, result, tick,
message, and worktree event is recorded there. The visualizer reads the
journal by fetching the `journal` branch of the garden repo and parsing
entry frontmatter (ts, kind, role) and the first paragraph of body text.

Also consumed: the bulletin board at `journal/README.md` for curated
summary sections (recent engagements, pending reviews, PR backlog, open
monitors). The bulletin is regenerated by the steward each cycle; the
visualizer renders it as-is alongside the live timeline.

### Source B: Worktree index (git, journal branch)

The worktree index at `journal/worktrees/<host>/<name>.md` records every
garden-managed checkout across every host (path, repo, branch, role,
status, heartbeat). The visualizer fetches these files from the `journal`
branch to show which worktrees are active, idle, or collected on each
machine.

### Source C: Schedule tree (git, journal branch)

The schedule at `journal/schedule/<owner-or-garden>/<trigger>--<id>.md`
lists pending and fired timed events. The visualizer reads pending events
and shows the next fire time, recurrence rule, and target role.

### Source D: Standing monitor logs (filesystem, /tmp)

The bash poll daemons write to `/tmp/garden-monitor-<slug>.log` and
`/tmp/garden-review-queue.log` (with companion `.pid` and `.err` files).
The visualizer tails these files to show recent GitHub events detected by
each daemon and to confirm daemon liveness (PID present, log updated
within the last 2 heartbeat periods).

### Source E: Active dispatch roots (filesystem)

Active dispatch sessions live under `<garden-root>/dispatches/<role>--<short-id>/`.
The visualizer lists these directories to show currently running subagents.
It does not read into the worktrees themselves (that would be invasive);
it uses the directory name and modification time as a presence indicator,
cross-referenced with the journal's most recent `dispatch` entry for
that short-id.

### Source F: GitHub API (optional, via `gh`)

For PR state enrichment, the visualizer calls `gh pr view <#N> --json
state,isDraft,mergeable,statusCheckRollup,reviews` on repos listed in the
journal bulletin board. This is optional and rate-limited. Without it, PR
references render as links only.

## Architecture

### Process model

A single long-running Python process that serves HTTP and keeps data
current via polling. No Node.js, no build step, no npm dependency. Python
is already present on the bot hosts (used by `monitor-poll.sh` for JSON
parsing) and the standard library `http.server` module is sufficient for
the initial version.

The server registers itself as a new standing process alongside the
monitor daemons. Its lifecycle mirrors the steward watcher daemon:

```
./garden visualizer start     # daemonizes, writes PID to /tmp/garden-visualizer.pid
./garden visualizer stop      # kills by PID file
./garden visualizer status    # checks PID / port / log freshness
./garden visualizer restart
```

### How it stays current

The server runs three background poll loops, each on its own thread:

| Thread | Data source | Poll interval | Mechanism |
|--------|-------------|---------------|-----------|
| Journal poller | journal branch (git) | 10 seconds | `git fetch origin journal`, then re-read `entries/` tree and `README.md`. Uses cursor tracking: records the last-seen entry path per role so subsequent polls only parse new files. |
| Filesystem poller | dispatch roots, monitor logs | 5 seconds | `os.listdir(dispatches/)` for active sessions; reads last line of each monitor `.log` file for event content; `os.stat()` for mtime/size. Tracks PIDs from `.pid` files. |
| GitHub poller | PR state via `gh` | 60 seconds | Calls `gh pr view` for each PR slug found in the bulletin. Caches results with a TTL. Skips if `gh` is not installed or rate-limited. |

The poll intervals target "good enough for a glance" rather than real-time.
The maintainer opens the page, sees data at most 10 seconds stale, and
finds cause for concern within 60 seconds.

### Poll journal frequency

Why git fetch every 10 seconds and not inotify? The journal is on a git
branch written by other agents on this machine and potentially on remote
machines. inotify only sees local filesystem events and only within a
single worktree. A `git fetch` followed by reading new entry files is
reliable across every writer regardless of location. Ten seconds is
conservative for a process whose total I/O per poll is a handful of small
files.

### Markdown rendering with stdlib only

The bulletin view and entry previews render Markdown as HTML. Python's
standard library has no Markdown parser, so the server ships a minimal
renderer (~60 lines) that handles the subset used in garden journal entries:

| Element | Syntax | Render |
|---------|--------|--------|
| Heading | `## text` | `<h2>text</h2>` (levels 2–4) |
| Unordered list | `- item` | `<ul><li>item</li></ul>` |
| Ordered list | `1. item` | `<ol><li>item</li></ol>` |
| Code block | ```` ``` ```` | `<pre><code>...</code></pre>` |
| Inline code | `` `code` `` | `<code>code</code>` |
| Link | `[text](url)` | `<a href="url">text</a>` |
| Bold | `**text**` | `<strong>text</strong>` |
| Horizontal rule | `---` | `<hr>` |

Not supported (and not present in journal entries): tables, blockquotes,
nested lists, image embeds, HTML passthrough. The renderer is
single-pass, regex-based, and safe by construction (HTML-escapes all text
content before inserting tags). Frontmatter (`---...---`) is stripped
before rendering.

### SSE for live push

Each poll loop emits events to an internal PubSub bus. Long-lived HTTP
connections (the browser's EventSource) subscribe to the bus for their
view. When the journal poller detects new entries, it pushes a
`journal:new-entry` event; the browser's timeline view appends the entry
without a full page reload.

### Framework choice

Python 3 standard library `http.server` for the MVP. Rationale:

- Zero new runtime dependencies. The garden currently has no Node.js
  dependency, and adding one for a dashboard is disproportionate.
- The visualizer is a maintainer tool, not a customer-facing service. If
  it serves a few hundred requests per day, `http.server` is adequate.
- If it grows beyond MVP (auth, WebSocket, charts), the architecture is
  structured so the HTTP layer can be swapped for Flask or FastAPI
  without changing the poll loops or template rendering.

Templates use Python f-strings with HTML escaping. No template engine.
Pages are small (one or two screens each) and the maintainer audience
means maintainability beats abstraction.

## Endpoints / views

The main page. No navigation bar; the user arrives here and the three
panels are on one scrollable page.

### Timeline (GET /timeline?since=1h)

A chronological feed of all journal entries, newest first:

```
+------------------------------------------------------------------+
|  TIMELINE  past 1h  [ 1h | 4h | 24h | 7d ]    (polling: 10s)    |
+------------------------------------------------------------------+
| 23:14:05  result  builder 4a8df9                                  |
|           Gap report for endojs/endo-but-for-bots#138: 17 gaps    |
|           surfaced in OCapN/Daemon transports design.             |
|           [ref: entries/2026/05/17/231405Z-result-builder-4a8df9] |
+------------------------------------------------------------------+
| 23:12:04  result  liaison 2757e1                                  |
|           Ferried #258 upstream as endojs/endo#3264.             |
|           [ref: entries/2026/05/17/231204Z-result-liaison-2757e1] |
+------------------------------------------------------------------+
| 22:58:02  dispatch  steward da0309                                |
|           Dispatch root: dispatches/steward--da0309/              |
|           Task: per-cycle inbox drain and bulletin refresh.       |
|           [ref: entries/2026/05/17/225802Z-dispatch-steward-da0309]|
+------------------------------------------------------------------+
```

The time range selector (1h / 4h / 24h / 7d) hides entries older than the
selected window. Each entry shows kind, role, short-id, body preview, and
a link to the raw entry file on the journal branch. Dispatch entries link
to the corresponding dispatch root directory (if still present).

Polled entries push in via SSE; if the browser is on the timeline view and
a new entry arrives, it appends to the top with a subtle highlight
animation. The view also shows PR state changes when the GitHub poller
detects them (e.g. "PR #226 transitioned from CHANGES_REQUESTED to
APPROVED").

`journal/worktrees/<host>/*.md` cross-referenced with filesystem presence.
Daemon health reads PID files and log mtimes. Schedule reads
`journal/schedule/<owner-or-garden>/*.md` pending events.

### Bulletin (GET /bulletin)

Renders `journal/README.md` as rendered Markdown. This is the maintainer
dashboard. The visualizer fetches the current content from the journal
branch and serves it as an HTML page with the same section hierarchy the
raw file has. Links to GitHub PRs and journal entries are clickable.

This view exists so the maintainer does not need to `git pull` the journal
branch and open the Markdown file locally.

## Sample visualization (concrete example)

The maintainer is away from the terminal. They open
`http://garden-host:2804/` on their phone. The dashboard shows:

- **Active dispatches**: one row "designer 299699 Expand garden-visualizer
  design doc" with age 3 minutes.
- **Recent results**: a builder's gap report on PR #138 from 4 minutes
  ago, and a liaison ferry result from 7 minutes ago.
- **Running monitors**: two green indicators for `garden-of-bits` and
  `jesc24`, one yellow for `review-queue` (last event 140s ago, slightly
  over the 120s cadence).

The maintainer taps "PR #138" in the bulletin panel. The visualizer opens
the GitHub URL. The maintainer sees the gap inventory and decides the
builder's findings need a fixer dispatch.

Without the visualizer, the maintainer would need to `ssh` in, `ls
dispatches/`, `tail /tmp/garden-monitor-*.log`, `git log journal`, and
open `journal/README.md` locally. The visualizer collapses these five
steps into one browser tab.

## Integration with the garden lifecycle

### Starting and stopping

The `garden` script gains a `visualizer` subcommand:

```
./garden visualizer start     # daemonize, write PID
./garden visualizer stop      # SIGTERM via PID file
./garden visualizer status    # print port/PID/stale-age
./garden visualizer restart
```

Internally, `garden visualizer start` runs:

```sh
nohup python3 <garden-root>/tools/garden-visualizer/server.py \
  --port 2804 \
  --garden-root <garden-root> \
  --host 0.0.0.0 \
  >> /tmp/garden-visualizer.log 2>&1 &
echo $! > /tmp/garden-visualizer.pid
```

The server binds to `0.0.0.0` so any container-local IP address serves
the dashboard. The garden runs in a container that only authorized
clients can access, so no further access control is needed at this layer.

### Cron / steward integration

The visualizer does not need cron or steward awareness. It is a
standalone process that reads filesystem and git state. It does not write
anything, so it cannot corrupt the journal. If it crashes, the garden
continues normally; the maintainer notices the lack of dashboard and
restarts it.

If the visualizer is down for more than one steward cycle, the steward
may detect the missing PID file and emit a journal `message` entry. That
is a future refinement.

### Dependency footprint

The server requires only the Python 3 standard library. Verified imports:

- `http.server` (HTTPServer, BaseHTTPRequestHandler)
- `json`, `threading`, `subprocess`, `os`, `time`, `pathlib`, `html`,
  `re` (for the minimal Markdown renderer), `collections`.

No `requirements.txt`. No `npm install`. No `pip install`. The server is
<500 lines of Python.

## Open questions

1. **Port.** 2804 proposed (mnemonic: 28 = garden-path letter count).
   Should it be configurable via `GARDEN_VISUALIZER_PORT` env var?

2. **Log rotation.** The server writes to
   `/tmp/garden-visualizer.log`. Logs in `/tmp` are lost on reboot,
   which is acceptable. If the server crashes and is restarted, old log
   content is overwritten (append mode: `>>`). Should the log rotate by
   size?

3. **Multiple hosts.** The initial implementation serves one host's view
   (the machine it runs on). The journal may carry worktree entries from
   other hosts. Those entries are still shown in the worktree table from
   the journal's worktree index, but dispatch roots and monitor logs
   from other hosts are invisible because they are not in this machine's
   filesystem. A future version could aggregate across hosts by reading
   each host's `journal/worktrees/<host>/` and pinging each host's
   visualizer endpoint for cross-host data.

4. **Process supervision.** On a containerized deployment, the container
   runtime restarts the visualizer if it exits. On bare metal, the
   maintainer runs `./garden visualizer start` manually or includes it
   in a systemd unit. Should the garden script install a systemd user
   unit?

## Implementation plan

Ordered by dependency:

1. **Poll loops.** Implement the three background threads (journal,
   filesystem, GitHub) as Python classes with a shared PubSub bus.
   Write unit-testable core with no HTTP awareness.

2. **HTTP server.** Single-file server with four routes (`/`,
   `/timeline`, `/state`, `/bulletin`). Templates inline as
   f-strings. SSE endpoint at `/events` for live push.

3. **Integration.** Add the `visualizer` subcommand to `garden` script.
   Write PID management.

4. **Tune.** Set poll intervals, test under concurrent journal writes,
   verify stale-age indicator thresholds.

Target total code: under 600 lines of Python.
